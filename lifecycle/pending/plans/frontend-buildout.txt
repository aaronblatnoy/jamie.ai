------------------------------------------------------------------------
OVERVIEW
------------------------------------------------------------------------

Build out the React frontend from scratch. frontend/src/ is currently
empty — only index.html, package.json, and vite.config.js exist.

The app targets finance students with no assumed technical knowledge.
Tone throughout: professional, direct, reassuring. Think "filling out
a brokerage account form" — familiar, clear, no jargon surprises.

Critically: users bring their own Anthropic and ElevenLabs API keys.
The app owner pays nothing for AI costs. Key setup must be approachable
for non-technical people who have never used an API before.

Stack:
  - React 18 + Vite (already scaffolded)
  - React Router v6 (URL-based routing — replaces earlier state-machine
    approach to support deep-links, back-button, and auth redirects)
  - Plain CSS with CSS custom properties (no Tailwind — keeps dep count
    minimal; dark theme via custom props)
  - canvas-confetti (key verification celebration)
  - No Redux, no Zustand — React context + useReducer

Full app flow:
  Landing → Sign Up / Sign In → API Key Setup (if keys not set)
  → Mode Select → Session Config → Interview → Feedback → Summary

Auth: JWT session token in localStorage. All API calls include
  Authorization: Bearer {token} header. Each user's API keys are
  stored server-side encrypted; the backend decrypts them on-the-fly
  when proxying Claude and ElevenLabs calls. The frontend never
  handles raw API keys except during the setup wizard form inputs.

Backend API is assumed on localhost:3001; Vite's proxy config already
forwards /api/* there.

PHASE 2 — AUTH CONTEXT, useAuth HOOK, API CLIENT
------------------------------------------------------------------------

Goal: centralize auth state, provide a typed API client that includes
the Bearer token automatically, and bootstrap session state on page load.

2.1  Create frontend/src/context/AuthContext.jsx

     Context shape:
       { user, token, isLoading, login(token, user), logout() }

     user shape (mirrors what backend returns on login/register/me):
       { id, email, hasAnthropicKey, hasElevenLabsKey,
         anthropicKeyVerifiedAt, elevenLabsKeyVerifiedAt }

     <AuthProvider> behavior on mount:
       1. Read token = localStorage.getItem('jamie_token')
       2. If token: call GET /api/auth/me with Authorization: Bearer {token}
            - 200 → set { user, token } in state; isLoading = false
            - 401 / network error → clear localStorage; isLoading = false
       3. No token → isLoading = false immediately

     login(token, user):
       localStorage.setItem('jamie_token', token)
       Set { user, token } in state

     logout():
       POST /api/auth/logout (best-effort; don't block on failure)
       localStorage.removeItem('jamie_token')
       Set { user: null, token: null } in state
       navigate('/')

     While isLoading=true, render a full-screen spinner to avoid
     layout flash (prevents flickering between Landing and /app).

2.2  Create frontend/src/hooks/useAuth.js
     Re-export useContext(AuthContext) with a guard:
       if context is undefined, throw Error('useAuth must be inside AuthProvider')

2.3  Create frontend/src/lib/api.js
     apiFetch(path, options = {}):
       - Always uses relative path (no hard-coded host — Vite proxy handles it)
       - Reads token from localStorage
       - Merges Authorization: Bearer {token} if token present
       - Sets Content-Type: application/json if body present
       - Returns { ok, status, data } — never throws on 4xx/5xx;
         returns { ok: false } so callers can branch cleanly
       - On network failure: returns { ok: false, status: 0, data: null }

     Named wrappers:
       apiGet(path)           → apiFetch(path, { method: 'GET' })
       apiPost(path, body)    → apiFetch(path, { method: 'POST', body: JSON.stringify(body) })

2.4  Required backend routes (new — not in backend-buildout.txt):
     POST /api/auth/register { email, password }
       → 201 { token, user }
       → 400 invalid input
       → 409 email already exists

     POST /api/auth/login { email, password }
       → 200 { token, user }
       → 400 invalid input
       → 401 wrong credentials

     GET  /api/auth/me  (Bearer)
       → 200 { user }
       → 401 invalid/expired token

     POST /api/auth/logout  (Bearer)
       → 200 { ok: true }

     POST /api/auth/password { currentPassword, newPassword }  (Bearer)
       → 200 { ok: true }
       → 401 wrong current password

     POST /api/user/keys/verify { anthropicKey?, elevenLabsKey? }  (Bearer)
       → 200 { anthropic: { valid, error? }, elevenLabs: { valid, error? } | null }
       Action: test each key with a minimal API call; encrypt + store valid
       keys server-side; update user hasAnthropicKey, hasElevenLabsKey,
       and *KeyVerifiedAt timestamps.

     Note: existing /api/evaluate and /api/tts routes must also be updated
     to decrypt and use the authenticated user's stored keys rather than
     .env keys — each user's session uses their own keys.

     User model additions needed in backend:
       email                 TEXT UNIQUE NOT NULL
       password_hash         TEXT NOT NULL
       anthropic_key_enc     TEXT (encrypted at rest)
       elevenlabs_key_enc    TEXT (encrypted at rest, nullable)
       has_anthropic_key     BOOL DEFAULT false
       has_elevenlabs_key    BOOL DEFAULT false
       anthropic_key_verified_at   TIMESTAMP
       elevenlabs_key_verified_at  TIMESTAMP

PHASE 7 — INTERVIEW SCREEN
------------------------------------------------------------------------

Goal: the core experience loop. Fetch question → speak it → user
answers → evaluate → show feedback → loop.

File: frontend/src/screens/InterviewScreen.jsx
Child components:
  frontend/src/components/interview/QuestionCard.jsx
  frontend/src/components/interview/JamieBlob.jsx
  frontend/src/components/interview/PushToTalkButton.jsx
  frontend/src/components/interview/TranscriptPreview.jsx
  frontend/src/components/interview/FeedbackPanel.jsx
  frontend/src/components/interview/SessionProgress.jsx

7.1  Session state (useReducer in InterviewScreen):
     {
       questions: [],      // completed questions
       currentIndex: 0,
       askedIds: [],       // for &exclude= param
       results: [],        // { question, answer, evaluation } per Q
       mode, config,
       phase: 'LOADING'|'SPEAKING'|'IDLE'|'LISTENING'|'PROCESSING'
              |'FEEDBACK'|'DONE'
     }

     On mount:
       - Load config from sessionStorage('jamie_session_config')
       - Fetch first question: GET /api/question?mode=X[&category=Y,Z][&exclude=]
       - phase → SPEAKING (or IDLE if voice disabled)

7.2  Question phase
     QuestionCard: large readable question text.
     SessionProgress: "Question N of M" progress bar at top.

     If voice enabled (config.voiceEnabled + user.hasElevenLabsKey):
       Auto-call POST /api/tts { text: question.question }
       Play returned audio/mpeg via Audio() with object URL
       Show subtle "Jamie is speaking..." in JamieBlob (state='speaking')
       phase = SPEAKING → on audio end → phase = IDLE

     If voice disabled: phase goes straight to IDLE on question load.

7.3  Recording phase
     PushToTalkButton:
       Modes: config.inputMode 'hold' or 'tap'
       Hold mode:  mousedown/touchstart → start SpeechRecognition;
                   mouseup/touchend → stop
       Tap mode:   click → toggle start/stop
       Spacebar:   same as primary interaction in both modes
                   (preventDefault to stop page scroll; track keydown
                   vs auto-repeat with a held flag)

       SpeechRecognition setup:
         continuous: false (or true for hold-mode — handle onend)
         interimResults: true
         lang: 'en-US'

       Unsupported (non-Chrome/Edge): show textarea fallback with
         [Submit Answer] button.

     TranscriptPreview: live interim transcript as user speaks.
     JamieBlob: state='listening' while recording.

7.4  Evaluation phase
     On transcript finalized:
       - phase = PROCESSING
       - Disable PTT
       - JamieBlob state='processing'
       - POST /api/evaluate {
           question: question.question,
           user_answer: transcript,
           mode,
           key_points: question.key_points,     // IB/RX
           star_hints: question.star_hints       // behavioral
         }
       - On response: store result; phase = FEEDBACK

7.5  FeedbackPanel (inline below QuestionCard, slide-up transition)

     IB / RX mode:
       Score bar (score * 100%)
       ✓ Hit — green chips for each hit key point
       ✗ Missed — red chips for each missed key point
       Feedback line (italic, muted): the feedback_line from Claude

     Behavioral mode:
       STAR breakdown — 4 rows:
         Situation / Task / Action / Result, each with a 0–3 star rating
         (filled ★ vs outline ☆, 3 stars per row)
       Overall score bar
       Strengths: green bullet list
       Improvements: amber bullet list
       Feedback line (italic)

     If voice enabled: auto-play POST /api/tts { text: feedback_line }
       after panel appears. Keep it to feedback_line only (concise).
       Show ✕ button to skip TTS playback.

     [Next Question →]  btn-primary
       - Add current question id to askedIds (exclude list)
       - If currentIndex + 1 >= config.count → navigate('/app/summary',
           { state: { results, mode, config } })
       - Else: fetch next question; clear FeedbackPanel; phase = SPEAKING

7.6  Controls (small, subtle — below PTT):
     [Skip]           — add to askedIds, do NOT record result,
                        do NOT increment completed count, fetch next
     [Repeat question] — re-play TTS for current question
                         (only visible if voice enabled)
     [End session]    — navigate to summary with current results

7.7  JamieBlob visual states:
     idle:       slow breathing pulse (CSS animation)
     speaking:   wave-bar animation (3-5 bars, varying heights)
     listening:  different hue + amplitude-reactive scale
                 (Web Audio AnalyserNode on mic MediaStream if granted;
                  fall back to generic pulse animation if denied)
     processing: shimmer/spinner animation
     Transitions: CSS transition on transform/opacity between states

        Full-bleed on mobile. QuestionCard + FeedbackPanel scroll;
        PTT button stays large (min 72px × 72px touch target).
        SessionProgress bar spans full width.

      Summary screen:
        Category table collapses to stacked rows on mobile.

10.2  Accessibility
      - All <input> elements have associated <label> via htmlFor/id.
      - Buttons have visible focus outlines (--color-accent, 2px solid,
        2px offset).
      - PTT button: aria-label="Hold to record your answer" (or "Tap
        to start recording" in tap mode).
      - aria-live="polite" region for transcript updates and phase
        state ("Recording...", "Evaluating...", "Jamie is speaking...").
      - Confetti canvas: aria-hidden="true" (purely decorative).
      - Color is NEVER the sole indicator: ✓/✗ glyphs accompany green/red.
      - Score bars: aria-valuenow, aria-valuemin, aria-valuemax.

10.3  Loading states
      Every screen that awaits data on mount:
        Show centered <Spinner /> + loading message.
        <Spinner />: a pure CSS rotating ring div (40px, 3px border,
        accent color, 0.8s linear spin). No GIF, no library.
        Loading messages:
          App bootstrap:    "Loading..."
          Question fetch:   "Getting your next question..."
          Evaluation:       "Evaluating your answer..."
          TTS in-flight:    Subtle "Jamie is speaking..." below blob,
                            not a full-screen block.

10.4  Error boundaries
      Wrap major screen groups in a React <ErrorBoundary> component
      that catches unhandled render errors and shows:
        "Something went wrong. Refresh the page to continue."
      with a [Refresh] button (window.location.reload()).

DESIGN DECISIONS & INVARIANTS
------------------------------------------------------------------------

D1. API keys never touch the frontend raw after submission.
    The PasswordInput masked field is the only place a raw key appears.
    After POST /api/user/keys/verify, the key is never returned or
    displayed again. The Settings screen shows only verified status, not
    the key value. This is both a security property and a UX property.

D2. URL routing (React Router) instead of state-machine routing.
    Earlier plan used a state-machine in App.jsx with no URLs. This
    version uses URL routing so the back button works, auth redirects are
    clean (navigate('/setup-keys', { replace: true })), and deep links
    like /app/interview are possible. The session state that can't live
    in the URL (results array) goes through location.state on navigate.

D3. No Tailwind — plain CSS only.
    Tailwind adds build complexity (PostCSS config, content glob purging,
    potential class-name collisions with third-party). For a focused app
    this size, custom properties + utility classes are more predictable
    and easier for future maintainers who don't know Tailwind.

D4. Voice is optional and gracefully degraded at two levels:
    Level 1: no ElevenLabs key → voice toggle hidden, text-only feedback.
    Level 2: SpeechRecognition unsupported (Firefox/Safari) → textarea
    fallback for answer input. The app is fully usable at Level 2.

D5. Session state in sessionStorage, not URL params.
    The session config (mode, categories, count, voiceEnabled, inputMode)
    is too large and contains arrays that don't serialize nicely into URL
    params. sessionStorage persists across navigations within the same tab
    but clears on tab close — appropriate for a practice session.

D6. Per-user API keys change the backend contract.
    The existing /api/evaluate and /api/tts routes were designed to use
    .env API keys. With per-user keys, the backend must decrypt the
    authenticated user's stored key and use it for each call. This is a
    backend concern, but frontend assumes it is handled transparently —
    the frontend just sends the Bearer token and expects the backend to
    route to the right key.
